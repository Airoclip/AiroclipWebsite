01
pillar 01 — no-code qa automation

qa automation tool — drag, drop,
automated in hours.

a drag-and-drop qa automation engine that lets anyone in the studio — designers, qa, producers — automate an entire unity game's test flows in hours, with zero code. built first for our own unity titles, with a generic core that already extends to other engines and projects with minimal adapter work.

// what it does

  • visual flow builder. compose taps, swipes, waits, asserts & branching by dragging nodes onto a canvas — no scripting required.
  • record & replay. captures full play sessions; flows generalize across builds and screen resolutions automatically.
  • ai-assisted authoring. describe an intent — "complete tutorial, then play level 3" — and the tool drafts the flow for you.
  • built-in assertions. ui state, scene state, and analytics events — every flow doubles as a regression test.

// why it matters

  • weeks → hours. a full game goes from weeks to automate to hours to automate; the qa and soak-testing cycle collapses with it.
  • no engineer-in-the-loop. non-technical team members own, edit, and maintain their own automation flows.
  • one flow, many uses. qa regression, performance soak tests, store-listing capture, ai-driven balancing runs — the investment compounds across teams.
  • engine-agnostic core. already extendable beyond unity to other game and app stacks — opens a second value stream for partner studios.

02
pillar 02 — ai centralized development framework

airoaiframework — one framework.
every team. shared rigor. live · in production

a portfolio of 30+ specialised ai agents covering the full mobile game-dev lifecycle — plan, build, review, ship, operate, learn. mobile-first and unity-tuned for zero-crash, zero-anr, no-microfreezes work; stack-agnostic across react, spring boot, and node for everything else. plain-english in, role-aware output that matches what each team expects. runs natively in claude code and openai codex.

// what makes it a portfolio, not a chatbot

  • specialised, not general. each agent has a single job, a checklist, an output format, and a verification step. products, not prompts.
  • role-aware. loads the right role — senior unity developer, architect, qa lead, product owner, troubleshooter — before it acts.
  • stack-aware. same agent behaves correctly whether the project is unity, spring boot, react, or node.js.
  • hooked, not hopeful. critical safety patterns (crash-trap sweep, policy-pack injection) run as deterministic python hooks with 56 unit tests between them — not ai promises.
  • memory-backed. projectmemory means every session starts smarter than the last.
  • verifiable output. structured blocks like trap sweep: and diff audit: that humans and downstream tooling can check.

// why it matters

  • senior judgment scales. a junior asking "fix this crash" gets the same trap sweep, regression-test, and verification rigor a 10-year veteran would apply.
  • cross-functional speed. product writes specs, designers map edge cases, qa generates test cases, devs ship code, support drafts replies — through one consistent ai layer.
  • two ai hosts, one framework. runs natively in claude code and openai codex — zero duplication, two delivery channels.
  • plain-english invocation. users describe the work; the framework routes the request, loads the right role, applies the right checklists.

03
pillar 03 — chat-driven analytics

conversation ai — ask in english.
get the answer from bigquery.

an intelligent analytics agent that lets anyone in the studio query our game and product analytics by simply chatting with it. wired live into bigquery and grounded in our actual analytics schemas — so a question in plain english comes back as a real, schema-correct result. now with on-demand a/b test results.

// what it does

  • natural-language interface. ask "d7 retention by country for last week's cohort" and get the answer in seconds — no sql required.
  • live bigquery connection. schema-aware over game analytics & product analytics — events, sessions, funnels, monetization.
  • intent → sql → result. the generated query is exposed for analyst review, audit, and reuse — never a black box.
  • multi-turn refinement. "now break it down by platform" · "compare to last month" — follow-ups refine without restarting.
  • on-demand a/b results. pull variant comparisons across retention, arpdau, funnels & segments the moment data is fresh — no data-team ticket required.

// why it matters

  • self-serve for non-analysts. pms, designers, producers, marketing — insight in seconds, not a ticket queue. decisions land inside the live-ops window.
  • schema-grounded, not hallucinated. every result traces back to a real bigquery query analysts can audit, verify, and lock as a reusable view.
  • foundation for live-ops automation. scheduled monitoring and closed-loop ramp/kill decisions are the natural next step — the hard part is solved.

04
pillar 04 — performance marketing ai

creative fatigue analysis —
spot it before it eats budget.

a performance-marketing ai layer that detects when ad creatives stop earning their spend and recommends rotation, refresh, or retirement before the channel does. campaigns stay fresh, spend stays efficient.

// what it does

  • time-series fatigue detection. watches ctr, cvr, roas & spend curves per creative; flags decay before the channel catches up.
  • per-platform & per-cohort. slices fatigue by audience, geo, placement & campaign — not just account-level averages.
  • refresh recommendations. outputs rotation windows and concrete "kill / hold / boost" calls automatically — every night across active campaigns.

// why it matters

  • less wasted spend. creative fatigue is the silent roas killer; catching it days earlier compounds across the portfolio — real money back in the budget every month.
  • data-led, not gut-led. replaces "this ad feels tired" with a concrete, defensible inflection point that marketing and finance can both stand behind.
  • always-on monitoring. alerts before performance dips show up in weekly reviews — the team acts on day 1 of decay, not day 7.
